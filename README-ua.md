# Розподілене Множення Матриць

Python система для обчислення `C = A × B` в **локальному режимі** (всі ядра процесора на одній машині)
та **розподіленому режимі** (власний TCP протокол або Ray на кількох вузлах),
з інтерфейсом в реальному часі для терміналу та веб, бенчмаркінгом та симуляцією мережі.

## Швидкий Старт

### Локальний режим

```bash
# Генерувати випадкову пару матриць 500×500 і помножити локально
python main.py --gen_random_matrixes yes --matrix_size 500

# З користувацькою прямокутною парою
python main.py --gen_random_matrixes yes --matrix_size_a 800x600 --matrix_size_b 600x400

# З збереженого файлу
python main.py --input_file data/matrices.npz

# Зберегти результат у файл
python main.py --gen_random_matrixes yes --matrix_size 1000 --output_file data/result.npz
```

### Розподілений режим — власний TCP бекенд

**Крок 1 — запустити робочі вузли на кожній віддаленій машині:**
```bash
# На робочій машині 1 (наприклад 192.168.1.2):
python worker.py --port 9000

# На робочій машині 2 (наприклад 192.168.1.3):
python worker.py --port 9000
```

**Крок 2 — запустити з головного вузла:**
```bash
python main.py \
  --remote_servers 192.168.1.2:9000 192.168.1.3:9000 \
  --gen_random_matrixes yes \
  --matrix_size 2000 \
  --backend custom
```

### Розподілений режим — Ray бекенд

```bash
# Спочатку запустити Ray на кожній машині:
#   Головна:   ray start --head --port=6379
#   Робоча: ray start --address=<HEAD_IP>:6379

python main.py \
  --remote_servers 192.168.1.2:6379 \
  --gen_random_matrixes yes \
  --matrix_size 2000 \
  --backend ray
```

### Веб інтерфейс

```bash
python main.py --gen_random_matrixes yes --matrix_size 1000 --ui web --web_port 8080
# Браузер автоматично відкриється за адресою http://localhost:8080
```

## Розподілений Алгоритм

```
Матриця A (M×K)  поділена на N горизонтальних блоків: A_0, A_1, … A_{N-1}
Матриця B (K×N)  повністю надсилається кожному робочому вузлу

Робочий вузол i обчислює:  C_i = A_i × B

Головний вузол збирає:  C = vstack(C_0, C_1, … C_{N-1})
```

- **N** = кількість доступних робочих вузлів
- Робочі вузли виконуються одночасно; головний збирає через `asyncio.gather`
- Порядок зберігається через індекс — сортування не потрібне

## Власний TCP Протокол

Власний бекенд використовує мінімальний бінарний протокол фреймування через asyncio TCP:

```
┌──────────────┬──────────────────────┬─────────────────────┐
│  msg_type    │  payload_length      │  payload            │
│  (4 байти)   │  (8 байтів, big-end.)│  (N байтів, msgpack)│
└──────────────┴──────────────────────┴─────────────────────┘
```

| Тип | Напрямок | Значення |
|------|-----------|---------|
| `MSG_TASK (1)` | Головний → Робочий | Блок матриці для множення |
| `MSG_RESULT (2)` | Робочий → Головний | Обчислений блок результату |
| `MSG_PING (3)` | Головний → Робочий | Перевірка здоров'я |
| `MSG_PONG (4)` | Робочий → Головний | Підтвердження живості |
| `MSG_ERROR (5)` | Будь-який | Повідомлення про помилку |

Корисне навантаження серіалізується **msgpack** (повертається до `pickle` якщо msgpack недоступний).
NumPy масиви передаються як сирі байти з метаданими форми і типу.

### Тестові Сценарії

| Сценарій | Конфігурація Мережі | Очікуване |

## Симуляція Мережі

Використовує Linux `tc` (Traffic Control) + `netem`. Потрібен `iproute2` і зазвичай `sudo`:

```bash
# Застосувати вручну:
sudo tc qdisc add dev eth0 root tbf rate 1mbit burst 10kb latency 50ms
sudo tc qdisc add dev eth0 parent 1:1 handle 10: netem loss 5%

# Видалити:
sudo tc qdisc del dev eth0 root
```

Або використовувати Python обгортку:
```python
from network.simulator import NetworkSimulator, PROFILES
sim = NetworkSimulator(interface="eth0")
sim.apply_profile("low_bandwidth")   # 1 Мбіт/с
# ... запустити бенчмарк ...
sim.clear()
```

Доступні профілі: `high_bandwidth`, `low_bandwidth`, `very_low_bandwidth`,
`unstable`, `worst_case`.

## Метрики

| Метрика | Опис |
|---|---|
| **Загальний Час** | Час реального часу від початку до збірки фінальної матриці |
| **Час Обчислень** | Чистий час `A_block @ B` (сумований по робочих вузлах) |
| **Час Збірки** | Час для `vstack` часткових результатів на головному вузлі |
| **Накладні Витрати Комунікацій** | Час Серіалізація + Передача + Десеріалізація |
| **Байтів Відправлено** | Загальна кількість байтів відправлених робочим вузлам |
| **Байтів Отримано** | Загальна кількість байтів отриманих від робочих вузлів |
| **Коефіцієнт Прискорення** | `T_локальний / T_розподілений` (спочатку встановити базовий) |
| **Час Робочих Вузлів** | Індивідуальний час обчислень на робочий вузол |

## Очікувана Продуктивність

| Розмір Матриці | Локально (2 ядра) | Розподілено 2-вузли (LAN) |
|---|---|---|
| 200 × 200 | ~0.01 с | ~0.05 с (накладні витрати домінують) |
| 500 × 500 | ~0.05 с | ~0.10 с |
| 1000 × 1000 | ~0.3 с | ~0.2 с (починає окупатися) |
| 2000 × 2000 | ~2 с | ~1.1 с |
| 5000 × 5000 | ~30 с | ~16 с |
| 10000 × 10000 | ~240 с | ~125 с |

> Розподілення окупається тільки коли час обчислень >> накладні витрати комунікацій.
> Для малих матриць (<1000×1000) на швидкій LAN, локальний режим зазвичай швидший.
